# 项目开发计划

## 项目概述

- **项目名称**: PanoDB (HarmonyOS PC 数据库管理工具)
- **项目类型**: HarmonyOS PC 原生应用
- **开发平台**: OpenHarmony 6.0 (API Level 20)
- **目标用户**: 开发人员、数据库管理员、数据分析师

## 1. 项目背景与目标

### 1.1 项目背景

随着鸿蒙生态的发展，HarmonyOS PC 用户对原生数据库管理工具的需求日益增长。目前 HarmonyOS PC 平台上缺乏成熟的第三方数据库管理应用，现有的解决方案主要依赖命令行工具或需要借助其他设备。本项目旨在开发一款原生的、易用的数据库管理工具，填补这一生态空白。

### 1.2 项目目标

1. 开发一款基于 OpenHarmony 6.0 (API 20) 的 PC 原生数据库管理应用
2. 支持 SQLite 数据库的创建、浏览、查询、编辑等核心功能
3. 提供桌面级的用户体验（自由窗口、多窗口）
4. 参考开源项目 ByteDance rdbStore 的设计模式，确保代码质量

## 2. 项目范围

### 2.1 建设规模

| 指标 | 预估 |
|------|------|
| 代码行数 | 约 15000 行 (ArkTS + C++ NAPI) |
| 页面数量 | 4-5 个主要页面 |
| 功能模块 | 6 个核心模块 |
| 研发周期 | 约 12 周 |

### 2.2 主要建设内容

1. **NAPI SQLite 引擎层**: C++ 实现原生 SQLite 操作接口
2. **ArkTS 服务层**: 数据库服务、Schema 服务、查询服务
3. **ArkUI 界面层**: 架构树、SQL 编辑器、数据表格、结果面板
4. **窗口管理系统**: 自由窗口配置、多窗口管理

## 6. 风险识别与应对 【已修订】

| 风险 | 可能性 | 影响 | 应对措施 |
|------|--------|------|----------|
| **应用沙箱隔离导致无法直接访问目标应用数据库** | **高** | **高** | **实现"文件导入/导出"机制：用户通过系统选择器选择.db文件，系统拷贝至本应用沙箱后操作，完成后导出回原位置** |
| NAPI 跨语言调用性能问题 | 中 | 中 | 预先进行性能测试，优化接口调用频率 |
| OpenHarmony API 变更 | 低 | 高 | 关注官方 release notes，及时更新 |
| 团队缺乏 HarmonyOS 开发经验 | 中 | 高 | 组织内部培训，安排技术分享 |
| RichEditor 语法高亮方案不可行 | 中 | 中 | 降级为 TextArea，延后至 Phase 2 评估 Web 组件方案 |
| TableView 大数据量性能问题 | 中 | 高 | 必须使用 LazyForEach + IDataSource，提前进行性能测试 |
